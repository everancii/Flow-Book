  <how-to-verify>
    1. Install the diagnostic APK built in Task 1 on a real Android device: `adb install build/app/outputs/flutter-apk/app-armeabi-v7a-release.apk` (or the ABI matching your device — arm64-v8a for modern devices).

    2. Clear the app log file so you start fresh: `adb shell run-as com.everancii.audiobookflow truncate -s 0 files/log/applogs.txt` (or delete and let the app recreate it).

    3. Open the FlowBook app, navigate to Sound-Books source, and open a Sound-Books book (tap a book from the browse list). Wait ~15 seconds. Do NOT press play manually — the test is whether auto-play works.

    4. Pull the log file: `adb shell run-as com.everancii.audiobookflow cat files/log/applogs.txt > /tmp/soundbooks-diag-1.txt`. Open the file and find lines containing `[DIAG]`.

    5. Record these data points for this book:
       - Checkpoint 1: processingState before setAudioSources (expected: idle or loading)
       - Checkpoint 2: did setAudioSources throw? (look for 'THREW' vs 'OK'); what processingState after?
       - Checkpoint 4: processingState + playing immediately after play()
       - Checkpoint 5 (500ms delayed): processingState + playing — if playing == false, note 'audioSession.setActive may have failed'; if playing == true but no audio heard, note 'native buffering drop'
       - Probe duration: time between checkpoint 1 and checkpoint 2 (approximate from log timestamps if available, or note 'fast'/'slow'/'hung')

    6. Repeat steps 3-5 for at least 2 MORE Sound-Books books (3+ total URLs). Try to vary: one popular book, one obscure book, one with many chapters if possible.

    7. Record the HYPOTHESIS VERDICT in the summary:
       - If checkpoint 5 shows playing == false across books → mechanism is audioSession.setActive failure (fork lines 1106-1127) → Phase 3 fix must address session activation, not buffering wait.
       - If checkpoint 5 shows playing == true but no audio → mechanism is native-side (ExoPlayer) buffering drop → Phase 3 ready-before-play await is the correct fix layer.
       - If checkpoint 2 shows setAudioSources THREW → mechanism is probe failure (404/corrupt MP3) → Phase 3 must surface the error (ERR-01).

    8. Record the TIMEOUT RECOMMENDATION:
       - If all probes complete under 2 seconds → 10s timeout is generous, keep it.
       - If any probe takes 5-10 seconds → 10s is borderline, consider 15s.
       - If any probe hangs past 10s → increase timeout default.

    9. Type "approved" with the verdict + data table, or describe what you found if it differs from the hypotheses above.
